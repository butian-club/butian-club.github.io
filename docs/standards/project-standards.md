# 项目规范

一套轻量的约定，目的是让项目可推进、可协作、可延续，而不是增加负担。

## 项目的生命周期

我们把每个方案 / 项目分为三种状态，与[项目列表](/projects)中的标记一致：

- **进行中（active）**：正在推进的方案。
- **已完成（completed）**：达成主要目标，产出可交付结果（如完整方案 + 答辩稿）。
- **归档（archived）**：暂停或终止，但保留资料供日后参考。

## 立项

一个新方案开始时，建议先写清楚三件事：

1. **目标**：这一阶段要产出什么？怎样算成功？最好是**可验证**的。
2. **范围**：这一阶段做什么、不做什么。
3. **分工**：对应哪些部门（管理 / 结构 / 人居 / 运营 / 基础设施），谁负责什么。

> 目标不必宏大，但一定要具体。「在 24 小时内产出一份覆盖结构、人居、运营的
> 金星轨道基地方案，并完成一次英文答辩」比「研究太空殖民」更适合作为一个明确目标。

## 推进

- **先读懂赛题（RFP）**：把需求与评分点逐条拆解成清单，作为方案的依据，避免跑偏。
- **留下记录**：关键决定、采用的参数、放弃的方案与原因都写进项目文档。
- **部门协同**：各部门的设计要互相对得上（如人居人口要和结构、生保、运营自洽）。
- **小步验证**：把目标拆成能检验的小步，每步都能看到结果。
- **定期同步**：在例会上同步进度、暴露阻塞、调整计划。

## 文档要求

每个方案至少应包含：

- **赛题解读**：需求清单与评分要点，以及我们对赛题的理解与假设。
- **方案说明**：目标、范围、当前状态，以及各部门的设计要点。
- **进展记录**：重要节点与阶段成果（也可发布到[活动记录](/blog)）。
- **复盘**：阶段或答辩结束时的回顾——做对了什么，下次改什么。

## 命名与归档

- 文件、目录命名清晰一致，避免「最终版_v2_真的最终」这类命名。
- 项目结束后，把资料按[资料归档](../archive.md)的约定整理留档。
