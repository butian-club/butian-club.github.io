# 账号状态

每个账号都有一个状态，决定它 **能不能登录**。由于注册采用 [邀请码制](./register.md)，
用有效邀请码注册的账号 **直接就是「已激活」**——没有「待审批」的中间态。

## 实际会用到的两种状态

| 状态 | 名称 | 能否登录 | 说明 |
| --- | --- | --- | --- |
| `ACTIVE` | 已激活 | 是 | 正常账号，可登录并使用其角色允许的全部功能 |
| `DISABLED` | 已禁用 | 否 | 曾经激活、后被管理员停用，无法登录 |

**流转**：

- **注册**（用有效邀请码）→ 账号 **直接 `ACTIVE`**。
- **禁用 / 恢复**（管理员在 [成员目录](https://gfssm.butian.club/members) 操作）：
  - `ACTIVE` →（管理员禁用）`DISABLED`
  - `DISABLED` →（管理员恢复）`ACTIVE`

```
  注册（有效邀请码）
        │
        ▼
   ┌─────────┐   管理员禁用   ┌──────────┐
   │ ACTIVE  │ ────────────▶ │ DISABLED │
   │ 已激活  │ ◀──────────── │ 已禁用   │
   └─────────┘   管理员恢复   └──────────┘

登录判定：ACTIVE → 正常进入平台；DISABLED → 不能登录
```

## 登录行为速查

| 状态 | 登录结果 |
| --- | --- |
| `ACTIVE` | 正常登录，进入工作台 |
| `DISABLED` | 拒绝登录 |

:::note[关于 PENDING / REJECTED]
数据模型里还保留着 `PENDING`（待审批）和 `REJECTED`（已拒绝）两个历史状态，它们来自早期的
「注册后审批」设计。**现在的邀请码注册流程不会再产生这两种状态**——有邀请码就直接激活，没有就无法注册。
你在日常使用中只会遇到 `ACTIVE` 和 `DISABLED`。
:::

:::tip[相关操作在哪里]
禁用 / 恢复账号由管理员在 [成员目录](./members.md) 完成；开不开放注册属于
[站点设置](./settings.md)；注册流程本身见 [注册与邀请码](./register.md)。
:::
